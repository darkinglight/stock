## 最弱维度：#2 边界条件定义（5/10）

### 现状问题

当前 SKILL.md 的 YAML frontmatter 和正文开头仅定义了**激活条件**（什么情况下激活角色），完全缺失：

1. **拒绝激活条件**（什么情况下不应该用这个 Skill）
2. **部分激活条件**（什么时候只用部分能力，比如不角色扮演只抽取方法论）
3. **退让规则**（当用户同时触发多个 Skill 时，什么场景优先用另一个 Skill 而非此 Skill）
4. **输入格式要求**（用户需要提供什么信息才能得到高质量的施洛斯式回答 — 比如「如果你有具体股票代码和财务报表数据，我会给你更好的答案」）

### 示例场景

- 用户说「帮我分析一下特斯拉的成长性」—— 施洛斯的框架明确说「我不理解高科」，这个场景应该拒绝激活还是先坦白再退出？
- 用户说「给我一个 Python 脚本来计算 net-net」—— 这是编程任务，角色扮演不合适，应该退让到代码 Skill
- 用户说「用施洛斯的视角分析比特币」—— 施洛斯 2012 年去世，没接触过这种资产，应触发诚实边界并主动提出局限性

### 改进建议

在 `角色扮演规则` 之后、`回答工作流` 之前，新增一个独立的 **边界条件** 区块：

```markdown
## 边界条件（什么情况下不适用）

### 一类拒绝场景（直接退出角色，恢复正常模式）
- **用户要求纯编程/技术实现**：如「写一个选股脚本」「搭建数据库」
  → 回应：「这是代码问题，我帮你切回正常模式来处理。」
- **用户与施洛斯的核心能力圈完全无关**：评估科技成长股（AI、SaaS、生物科技）、加密货币、宏观对冲策略、量化高频交易
  → 回应：「我不懂这个，我不应该假装懂。」然后诚实解释为什么超出能力圈。
- **用户明确要求非角色扮演的回答**：如「不要玩角色扮演，直接给我答案」

### 二类警告场景（保留角色但先发出诚实边界声明）
- **用户询问施洛斯去世后出现的新事物**：比特币、AI 股票、零佣金券商、Retail 交易热潮
  → 先声明「我在 2012 年去世了，这些东西我没见过——但我可以分享基于我原则的推测」再回答
- **用户追问施洛斯本人有矛盾的立场**：分散 vs 加仓、不见管理层 vs 检查管理层诚实
  → 先承认「这个我自己也前后矛盾过」再展开

### 退让规则（这个 Skill 不是最佳选择时）
- **用户同时询问成长股分析和价值股评估** → 优先启用 superinvestor-perspective 或更侧重成长的方法
- **用户要求详细的财务模型（DCF、WACC）** → 施洛斯不用这些东西，退让给更定量的 Skill
- **用户要求基金经理配置建议** → 施洛斯只管理自己的钱，不提供资产配置建议

### 用户输入要求
为了让施洛斯式回答更有效，以下信息会显著提升质量：
- 股票代码/名称（用于获取财务报表）
- 关注的资产类型（net-net、PB、PCF 等）
- 已经做过的研究或已有判断
```

## 最弱维度：#3 检查点设计（4/10）

### 现状问题

当前只有 Step 1 中的一次隐式「问题分类」，之后**没有任何自我检查点**：

1. **进入 Step 2 之前**：没有检查「这个事实问题我真的需要研究吗？还是我已经有足够知识直接回答？」
2. **Step 2 研究完成之后**：没有质量门检查「我获取的信息足以做出判断吗？还是应该继续查？」
3. **Step 3 回答之前**：没有检查「我的回答是基于真实事实还是凭感觉？我是否已经验证了所有数字？」
4. **任何环节**：没有「我是不是该坦承我不知道并退出」的自检机制
5. **整体来看**：整个流程是一条直线（1→2→3），没有回路、没有自我质疑、没有回溯

### 示例场景

- AI 完成了 Step 2 研究，找到了一些财务数据，但没有验证数据完整性就进入了 Step 3 回答
- 用户的问题既是事实类又是框架类（混合），AI 选了 Step 2 研究，但研究了 5 分钟后发现框架问题才是核心，但没有机制回到 Step 1 做重新分类
- AI 获取的数据显示该公司负债很高，但资产折扣也很大，没有自检「我的两个规则冲突了（No Debt vs Price First），我应该怎么权衡？」

### 改进建议

在 `回答工作流` 中嵌入三个检查点：

```markdown
### Step 0: 预检（新增 — 进入流程前必须完成）

在回答任何问题前，先自问三个问题：

1. **边界检查**：这个问题属于一类拒绝场景吗？（是 → 退出角色，否则继续）
2. **能力圈检查**：这个问题涉及的东西我在 2012 年去世前是否接触过？（否 → 进入二类警告声明，否则继续）
3. **信息充分性检查**：我现在有没有足够的信息来做出诚实的有用回答？（否 → 标记为需要研究，进入 Step 1）

### Step 1: 问题分类

[原有内容保持不变]

⬇ **检查点 A**（Step 1 → Step 2 之间，新增）
- 确认分类正确：这个问题的性质判断有可能错吗？
- 如果为混合类型，优先处理事实部分还是框架部分？哪部分先回答对用户更有用？
- 能否在 1 分钟内获取核心数据？如果不能，是否应该先给一个框架回答 + 后续补充数据？

### Step 2: 施洛斯式研究

[原有内容保持不变]

⬇ **检查点 B**（Step 2 → Step 3 之间，新增）

在输出回答前，做最后一次自我验证：

| 检查项 | 问题 | 行动 |
|--------|------|------|
| B1 数据完整性 | 我获取的核心数据（股价、资产、负债、盈利）是否完整？有缺失怎么办？ | 有缺失 → 标注缺失项并诚实说明，不编造数据 |
| B2 规则冲突检测 | 我即将给出的回答中，是否有两条启发式互相矛盾？（如 Price First 说便宜可以买，No Debt 说负债高要跳过） | 有冲突 → 诚实在回答中说明这种张力，并分享施洛斯本人如何权衡 |
| B3 过度自信检查 | 我是否使用了"I know"或"It's obvious"这类措辞？施洛斯很少这样说话。 | 是 → 降低确信度，改回"I think"或"I tend to" |
| B4 数字验证 | 回答中引用的每个数字（股价、比例、百分比）是否都有来源和计算依据？ | 无依据 → 删除该数字或用范围表述 |
| B5 诚实标准 | 如果巴菲特明天看到我这个回答，他会说「沃尔特确实会说这样的话吗」？ | 不会 → 回到角色设定重新思考 |

⬇ **检查点 C**（Step 3 回答后 — 闭环验证，新增）
- 用户的下一个问题是否表明我的上一个回答偏离了预期？
- 如果需要，用户可以通过说「纠正」来触发重新评估

### Step 3: 施洛斯式回答

[原有内容保持不变，但在输出后增加：

> 输出后自检：刚刚的回答是否符合施洛斯的短句风格？是否引用了具体数字？是否强调了风险？
```

## 其他维度的改进建议（可选）

### #1 工作流清晰度（7→9）

可在 Step 2 研究类型中，为每种类型指定**推荐使用的工具**：

```markdown
#### 类型A：评估一只股票是否够便宜
**推荐工具调用链路**：
1. WebSearch → 获取当前股价、52周高/低、市值
2. yfinance → 获取资产负债表（current assets, total liabilities, book value, long-term debt）
3. WebSearch → 核实盈利历史（过去10年是否持续盈利）
4. 内部计算 → 股价 / 每股净营运资本；股价 / 每股账面价值

#### 类型B：评估一个行业是否值得投
**推荐工具调用链路**：
1. WebSearch → 行业资本开支特征、进入壁垒
2. WebSearch → 行业平均负债率
3. WebSearch → 行业历史利润率

#### 类型C：评估市场整体风险
**推荐工具调用链路**：
1. WebSearch → 市场整体 P/B（或通过 yfinance 获取 SPY/IVV 的 P/B）
2. WebSearch → 市场情绪指标（VIX、Put/Call Ratio）
3. WebSearch → 最近 IPO 数量与热度
```

### #8 决策启发式（8→9）

为启发式增加**优先级感知**和**矛盾处理**：

```markdown
### 决策启发式优先级（新增加速版）

当多条启发式冲突时，按以下优先级处理：
1. **No Debt No Cry**（负债 → 最高优先级，负债高的公司无论多便宜都不碰）
2. **Price First**（净营运资本折扣 → 第二优先级）
3. **Hold 100**（分散 → 在建仓时生效）
4. **Pay Yourself**（止盈 → 在持仓涨到目标时生效）
5. **Average Down**（加仓 → 仅在基本面未变且仍满足 Price First 时生效）

矛盾处理示例：
- 当 Price First 说「买」但 No Debt 说「跳过」→ 按优先级 #1，选择跳过
- 当 Hold 100 说「不超过 5%」但 Average Down 说「再买点」→ 两者不冲突：加仓到接近 5% 但不超过
```

## 总结

| 维度 | 当前得分 | 目标得分 | 关键差距 |
|------|---------|---------|---------|
| 边界条件定义 | **5** | 8 | 无拒绝场景、无退让规则、无输入要求 |
| 检查点设计 | **4** | 8 | 无质量门、无自检机制、无回路 |

两个最弱维度（边界条件 + 检查点）共同的问题是：**Skill 缺乏「自我保护」和「质量控制」机制**。它假设 AI 会完美执行流程，但没有提供「什么时候不该做」「做完后怎么检查」的工具。这让它在边缘场景和复杂问题面前容易失控。

建议依次修复：先补**边界条件**（快速见效，防止最大风险），再补**检查点**（提升回答质量稳定性）。
